## 1. 需求回顾

一句话原始需求：**通过二维码动态翻页的单向文件传输套装**——Windows PC 发送端读取纯文本文件、分段生成二维码、动态翻页；手机接收端用摄像头连续识别、本地拼装还原、校验完整性，**数据全程不经过互联网**。

核心硬约束：
- **数据隐私**：接收端识别与文件内容全程手机浏览器本地处理（getUserMedia + jsQR + IndexedDB），公网 HTTPS 仅托管静态文件。
- **减少依赖**：发送端纯单文件 HTML+JS，**双击 file:// 即用**，不依赖 CDN；库源码置于本地 `lib/`；core 零依赖。
- 协议 `<T><P>|<PAYLOAD>`；近似字节分页（步长 ≥4，不切断 UTF-8 字符）；确定性可复现（同文件+同步长+同任务序号=相同二维码序列）；VMAX=10；SHA-256。

成功标准 SC1–SC5 为可自动化验证；SC6（浏览器双击 sender / 手机 HTTPS 扫描 receiver）为手动项。

## 2. 方案摘要

三块结构，core 与浏览器/Node 共用：

| 层 | 文件 | 职责 |
|---|---|---|
| core（零依赖 ESM） | `core/paginate.mjs` | 确定性 UTF-8 安全分页（MIN_STEP=4，不切断字符） |
| | `core/protocol.mjs` | `<T><P>\|<PAYLOAD>` 编解码 + **前导 BOM strip**（T2-C8，qrcodejs 中文注入 EF BB BF 的关键修复） |
| | `core/hash.mjs` | SHA-256（subtle 优先 + 纯 JS fallback，保 file:// 可用） |
| | `core/params.mjs` | V1–10 容量表 / maxStepBytes / recommendVersion / estimateTime |
| | `core/index.mjs` | buildPages / reassemble + 统一 re-export |
| sender | `sender.html` | **内联** core（file:// ESM CORS）+ qrcodejs；参数概要→确认→翻页→重传 |
| receiver | `receiver.html` | HTTPS `import` core + getUserMedia + jsQR + IndexedDB；覆盖式 set + meta.t 切换清空 + hash-gated 下载 |
| lib（vendored） | `lib/qrcode.min.js`（19.9KB）、`lib/jsQR.min.js`（251KB） | 本地第三方库（jsdelivr） |
| test | `test/*.test.mjs` ×7 | Node v20 `node --test`：单元 + e2e + 结构检查 |

**grilling 阶段（源码查证）已落实的关键修复**：① BOM strip（decodePage）；② BOM 占 3 字节 → sender `recommendVersion(step+3,…)`；③ 步长超 VMAX → analyze 禁用"确认开始"；④ receiver 覆盖式 set（抗误识别）；⑤ IndexedDB 按 meta.t 重置（clearDB）。已知限制：qrcodejs 用 `charCodeAt`，emoji（BMP 外代理对）会失败（纯文本无影响）。

## 3. 执行过程（subagent-driven，角色路由）

**并行可行性重评**：初始 SERIAL 评估后，识别到 **T1–T4 为互不依赖的 leaf 模块**（paginate/protocol/hash/params 文件集完全不相交、互不 import），改为 **PARALLEL wave**（同一工作树、不相交文件、各跑聚焦测试避免 full-suite 竞争；fixer 不触碰 ledger）。共 3 个并行 wave + 串行关键路径：

| Wave | Task | 角色 | 方式 |
|---|---|---|---|
| 1 | T1 paginate | fixer→review | 串行（首个） |
| 1 | T2 protocol / T3 hash / T4 params | 3×fixer 并行 → 3×oracle-rw 并行评审 | **并行** |
| 2 | T5 index+e2e（依赖 T1-4） | fixer→review | 串行 |
